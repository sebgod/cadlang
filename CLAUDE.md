# cadlang — instructions for Claude

A small declarative DSL for describing 3D CAD parts in Python. One
description emits both a watertight STL preview (via a real CSG kernel)
and a Fusion 360 parametric script. A STEP importer can reverse-engineer
existing CAD files into starter `.cad.py` descriptions.

cadlang is a general-purpose library. Real work is organised into
**projects** — directories that contain a `project.cadlang` manifest plus
the source files it references. `cadlang build` executes the manifest:
import STEPs, assemble mated parts, emit STL/Fusion outputs. See
`example-project/` for a working project. Multi-part assemblies are
described declaratively via a `<name>.yaml` that declares interfaces and
mates; parts position themselves by aligning interfaces, no hand-computed
poses needed.

Per-project notes (open questions, measurements, context for the specific
task) live in a `STATE.md` inside the project directory — **not** in this
file.

## Pipeline

```
            ┌───────────────────┐
            │  parts/foo.cad.py │   ← source of truth, hand-written or imported
            └─────────┬─────────┘
                      │
                      │ python parts/foo.cad.py
                      ▼
  ┌───────────────────────────────────────────────┐
  │ cadlang.py — Design + features + two backends │
  └──────┬─────────────────────────────┬──────────┘
         │                             │
         ▼                             ▼
  parts/foo.stl                  parts/foo_fusion.py
  parts/foo_preview.png                 │
  (manifold3d CSG → trimesh)            │
  (matplotlib sanity preview)           ▼
                             Fusion 360 Script: Run
                                → parametric body in timeline
                                → edit via Modify → Change Parameters
                                → re-running foo.cad.py stays the single
                                  source of truth; do not tweak
                                  numeric points inside the Fusion timeline
```

Reverse direction (optional):
```
  thing.step   ──►  stepimport.py  ──►  parts/<src>_partNNN.g.cad.py (starter)
                                         + full report on stdout
                                         edit → re-run → matches forward pipeline
```

## Folder layout

```
cadlang/
├── cadlang.py            # Design class + STL/Fusion backends + `cadlang build` CLI
├── stepimport.py         # STEP reader + feature recogniser + .cad.py emitter
├── assembly.py           # YAML assembly loader + mate solver + combined STL
├── CLAUDE.md             # this file — generic language reference
├── README.md             # user-facing docs
├── ROADMAP.md            # longer-term direction / planned features
├── FOLLOW_UP.md          # open questions / TODOs across the tooling
├── requirements.txt      # pinned minor versions
└── example-project/<name>/
    ├── project.cadlang        # manifest — `cadlang build` entry point
    ├── assembly.yaml          # (optional) multi-part assembly description
    ├── STATE.md               # per-project notes (measurements, decisions, TODOs)
    ├── <whatever>.step        # source imports (sit at the project root)
    ├── <assembly>.stl              ← generated by `cadlang build`
    ├── <assembly>_preview.png      ← generated
    └── parts/
        ├── <name>.cad.py                 # hand-written part
        ├── <name>.stl                     ← generated
        ├── <name>_preview.png             ← generated
        ├── <name>_fusion.py               ← generated
        ├── <src>_partNNN.g.cad.py        # auto-imported from STEP
        ├── <src>_partNNN.g.stl             ← generated
        ├── <src>_partNNN.g_preview.png     ← generated
        └── <src>_partNNN.g_fusion.py       ← generated
```

Outputs live next to the source `.cad.py`. Hand-written parts use
`<name>.cad.py` → `<name>.stl` / `<name>_preview.png` / `<name>_fusion.py`.
Imported-from-STEP parts use `<name>.g.cad.py` → `<name>.g.stl` /
`<name>.g_preview.png` / `<name>.g_fusion.py`. The `.g.` infix propagates
from source to outputs so generated and hand-written parts with the same
stem don't collide. Generated artifacts are disposable — regenerate by
running the `.cad.py`.

## The DSL

A part file is a Python script:

```python
import sys, pathlib
HERE = pathlib.Path(__file__).resolve().parent
sys.path.insert(0, str(HERE.parent))
from cadlang import Design, Rect, Circle, OffsetPlane, Circular

d = Design('MyPart', units='mm', params={
    'outer_r': 25,
    'height':  20,
})

# ... feature calls ...

if __name__ == '__main__':
    slug = 'my_part'
    d.emit_stl(str(HERE / f'{slug}.stl'),
               render_png=str(HERE / f'{slug}_preview.png'))
    d.emit_fusion(str(HERE / f'{slug}_fusion.py'))
```

### Core concepts

- **Design**: a named container with `params`, `units`, and an ordered list
  of `features`. Features run in order; later features operate on the result
  of earlier ones.
- **Expressions**: anywhere a dimension is accepted, pass a number (`5.0`)
  or a string that evaluates against `params` plus `math.*`
  (`'outer_r/2 + 1'`). Evaluated with `__builtins__` disabled — no side
  effects.
- **Two backends** from the same feature list:
  - `emit_stl(path, render_png=...)` — evaluates expressions to numbers and
    builds a watertight mesh via `manifold3d` CSG.
  - `emit_fusion(path)` — writes a Fusion Python script that recreates the
    part parametrically. `params` flow into Fusion `userParameters` so the
    dimensions stay editable in Fusion's Change Parameters dialog.

### Feature: revolve

Axisymmetric body from a closed 2D profile revolved around an axis.

```python
d.revolve(name='body', plane='XZ', axis='Z', profile=[
    ('inner_r', 0),
    ('outer_r', 0),
    ('outer_r', 'height'),
    ('inner_r', 'height'),
])
```

- `plane`: the sketch plane holding the profile. Currently only `'XZ'`.
- `axis`: revolution axis. Currently only `'Z'`.
- `profile`: list of `(u, v)` points where `u` is the radial coord in-plane
  and `v` is the axial coord. Polygon is auto-closed.

### Feature: extrude

Prismatic body from a closed 2D polygon extruded along the plane normal.

```python
d.extrude(name='plate', on='XY', height='thickness', profile=[
    (0, 0), ('length', 0), ('length', 'width'), (0, 'width'),
])
# …or with an offset plane:
d.extrude(name='step', on=OffsetPlane(base='XY', distance='z_base'),
          height='step_height', profile=[...])
```

- `on`: `'XY'` or `OffsetPlane(base='XY', distance=expr)`. Non-XY planes
  not supported yet.
- `profile`: list of `(x, y)` points in the sketch plane. Auto-closed.
- `height`: signed; positive extrudes along +plane-normal (+Z).

Multiple `revolve` or `extrude` features get unioned into one body.

### Feature: cut

Subtracts a prism from the current body. Sketch → extruded solid → boolean
difference.

```python
# (a) cut from the body's top face (axial Z plane at max profile v)
d.cut(name='slots',
      on=d.top_face('body'),
      sketch=[Rect(x=(0, 'a'), y=('-1', 1))],
      depth='-2',
      pattern=Circular(axis='Z', count=4))

# (b) cut from a YZ-offset plane (radial holes through a cylindrical wall)
d.cut(name='holes',
      on=OffsetPlane(base='YZ', distance='outer_r'),
      sketch=[Circle(center=(0, 'hole_z'), radius='hole_dia/2')],
      depth='-hole_depth',
      pattern=Circular(axis='Z', count=4))

# (c) cut from an XY-offset plane (axial through-holes in a plate)
d.cut(name='bolt_holes',
      on=OffsetPlane(base='XY', distance='plate_top_z'),
      sketch=[Circle(center=('hx1', 'hy1'), radius='hd/2'),
              Circle(center=('hx2', 'hy2'), radius='hd/2')],
      depth='-plate_thickness')
```

- `on`:
  - `d.top_face(body_name)` — the highest-Z face of a named revolve/extrude
    body
  - `OffsetPlane(base, distance)` with `base ∈ {'XY', 'YZ'}` (XZ not yet on
    STL)
- `sketch`: list of `Rect(x=(u0,u1), y=(v0,v1))` or
  `Circle(center=(u, v), radius=r)`. Use plane-local coords in `(u, v)`.
- `depth`: signed expression. Positive = along +plane-normal; negative =
  the other way. For a cut going into the body, this is usually negative.
- `pattern` (optional): `Circular(axis='Z', count=N, total_angle='360 deg')`.
  The entire sketch is replicated around the axis.

### Emitters

```python
d.emit_stl(path, render_png=None, n_seg=360)
d.emit_fusion(path)
```

- `n_seg` controls cylinder/revolve tessellation (default 360).
- `render_png` is optional; it produces a 3-pane matplotlib preview. For
  thin prismatic parts the renderer falls back to three iso angles instead
  of top/bottom, because matplotlib's painter's-algorithm 3D struggles with
  flat parts — if the preview looks off, **trust the STL** and check it in
  Windows 3D Viewer / Fusion.

## Feature coverage

| feature                                 | STL | Fusion | importer |
|-----------------------------------------|:---:|:------:|:--------:|
| `revolve(XZ, Z, profile)`               | ✅  | ✅     | ✅        |
| `extrude(on='XY', profile, height)`     | ✅  | ✅     | ✅ (bbox only, layered) |
| `extrude(OffsetPlane('XY', …))`         | ✅  | ✅     | ✅        |
| `cut(top_face, Rect, Circular)`         | ✅  | ✅     | ❌        |
| `cut(OffsetPlane('YZ'), Circle, Circ.)` | ✅  | ✅     | ✅        |
| `cut(OffsetPlane('YZ'), Circle)` (span) | ✅  | ✅     | ✅ (saddle) |
| `cut(OffsetPlane('XY'), Circle)`        | ✅  | ✅     | ✅ (axial bores) |
| Non-XZ revolves / non-Z axes            | ❌  | 🚧     | ❌        |
| Fillets, chamfers as ops                | ❌  | ❌     | ❌        |
| STL → cadlang importer                  | —   | —      | ❌        |

Unsupported features print a `cadlang[stl] skipping …` warning at emit
time; the Fusion script still includes them.

## STEP importer (`stepimport.py`)

```
python stepimport.py thing.step                        # just report
python stepimport.py thing.step -o parts/              # emit .cad.py per body
python stepimport.py thing.step -o parts/x.cad.py      # single-body output
python stepimport.py thing.step --name Foo -o parts/   # custom base name
```

What it handles:

- Parses AP203/AP214 STEP text into a record dict (no CAD kernel required).
- Walks each `MANIFOLD_SOLID_BREP` separately (one `.cad.py` per body).
- Classifies a body as `revolve` (has ID/OD cylinder pair + Z-axial) or
  `extrude` (has ≥2 Z-normal planes, not axisymmetric).
- For revolves: extracts ID/OD, height, and radial bore patterns (incl.
  circular-pattern count via opposite-pair heuristic).
- For extrudes: detects Z-levels from vertex Z-values → emits stacked
  layers (each a bbox rectangle); detects axial through-holes; detects
  lateral-axis cylinders (saddle/concave mating surfaces) and emits them
  as `cut(...)`.
- Measures hole depth from trim-circle centers, falling back to edge
  vertex-point projections onto the axis when trims are B-splines.

Assumes a single global coordinate frame — nested SHAPE_REPRESENTATION
transforms are not resolved yet. True for typical single-body Fusion
exports.

## How to add a new part

1. Copy an existing `.cad.py` as a starting template, or import a STEP
   file via `python stepimport.py thing.step -o parts/`.
2. Adjust `params` and features. The `.cad.py` is the source of truth —
   don't re-edit generated artifacts.
3. `python parts/<name>.cad.py` — regenerates `<name>.stl`,
   `<name>_preview.png`, `<name>_fusion.py` next to the source.
4. Sanity-check the preview. For flat / complex parts, open the `.stl` in
   Windows 3D Viewer for a cleaner render.
5. Open the `_fusion.py` in Fusion: **Utilities → Scripts and Add-Ins →
   Scripts → + Create → From Existing → Run**. The body lands in the
   timeline with editable parameters.

## Projects (`project.cadlang` + `cadlang build`)

A project is a directory containing a `project.cadlang` manifest and the
source files it references (STEP inputs, hand-written `.cad.py` files,
assembly YAMLs). All paths inside the manifest are resolved relative to
the directory containing `project.cadlang`.

```
cadlang build [PATH]        # PATH can be a dir, a project.cadlang file,
                            # or omitted (searches cwd upward)
```

Manifest schema:

```yaml
name: <project-name>
description: ...               # free text
steps:
  - import:
      source: thing.step       # .step file to read
      name: Thing              # base name for generated <name>_partNNN.g.cad.py
      out: parts/              # destination directory (default: parts/)
  - assemble: assembly.yaml    # run the mate solver on this YAML
  # more step kinds as needed
```

Each step is a single-key mapping; the key names the action. Steps run
strictly in order. Adding a new action is one `_do_<name>()` helper in
`cadlang.py` + a `_STEP_ACTIONS` entry.

## Assemblies (`assembly.py`)

```
python assembly.py path/to/<name>.yml                # emit combined STL + preview
python assembly.py path/to/<name>.yml --regenerate   # re-run each part first
```

Schema is fully documented at the top of `assembly.py`. Short version:

```yaml
name: <assembly-name>
units: mm

interfaces:              # per part TYPE (file-stem), list its mating features
  <part_type>:
    <iface_name>:
      kind: bolt_pair | tabbed_hole_stack
      # kind-specific fields (positions + axis, or radius/tab_count/hole_z)

parts:
  <slot>:
    source: ../parts/<name>.cad.py
    anchor: true          # exactly one part should be the assembly origin
    count: 4              # optional — multi-instance

mates:
  - part: <slot>           # which part to place
    from: <iface>          # OR {interface: ..., tab: ..., holes: [...]}
    to:                    # where on another part this one mates
      part: <other_slot>
      instance: 0          # default
      interface: <iface>
      tab: "{i}"           # {i} = instance index of the part being placed
      holes: [a, b]
```

The solver builds a dependency graph from mates, starts at `anchor`, and
computes rigid 4×4 transforms by aligning interface frames (bolt-pair
midpoint + axes). Legacy `pose:` / `circular:` per-part specs still work
for parts that don't declare interfaces.

## How to extend cadlang

Adding a feature kind (e.g. a new `pocket()` op):

1. Add a method on `Design` that appends `{'op': 'pocket', ...}` to
   `self.features`.
2. Update the STL backend (`_build_mesh`) — dispatch on `feat['op']`,
   build the tool manifold, union/subtract against `body`.
3. Update the Fusion backend (`_emit_fusion`) — add an `_emit_pocket(…)`
   helper and dispatch on `feat['op']`.
4. Update the feature-coverage table in this file.

Adding a new plane to an existing op: extend `_build_cut_tool` (STL) with
the new `on` branch, and make sure `_emit_cut` can render it in Fusion
(most plane types are already dispatched via `_PLANE_ATTR`).

## Environment

- Windows x64, Python 3.10. Just call `python`, never `python3`.
- `pip install -r requirements.txt` covers numpy, trimesh, matplotlib,
  manifold3d. The last one is the CSG kernel; without it no STL.
- Fusion 360 API internally uses cm. `cadlang` auto-converts numeric point
  coordinates mm→cm when emitting the Fusion script; expression strings
  with unit suffixes (`'10 mm'`) pass through unchanged.

## Invariants

- Every `params` value is a number in the design's `units` (always `mm`).
- Every expression string is `eval`-safe: `__builtins__` disabled, only
  `params` + `math.*` in scope.
- Profile polygons auto-close (don't repeat the first point at the end).
- Feature order matters: later features see the union/difference of earlier
  features.
- The first body-producing feature (`revolve` or `extrude`) defines the
  base body; subsequent ones union or subtract as their op requires.

## Sanity checks after changes to cadlang

1. `cd example-project/<any>` and run `python ../../cadlang.py build` — the
   whole pipeline (STEP import → per-part emit → assembly) should succeed
   end-to-end, and the combined STL should be watertight-ish (concatenated,
   not CSG-merged; see FOLLOW_UP).
2. Round-trip at least one hand-written part — re-run its `.cad.py` and
   confirm the STL is watertight and volume is in the ballpark of a hand
   calculation.
3. Open an `.stl` in Windows 3D Viewer to sanity-check geometry when the
   matplotlib preview is ambiguous.
